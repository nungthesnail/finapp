# AGENTS.md

Инструкции для AI-агентов, работающих в этом репозитории.

## 1. Источники правды

- Основной документ требований: `SPEC.md`.
- Если реализация расходится с `SPEC.md`, сначала предложить изменение `SPEC.md`, затем менять код.
- В спорных случаях приоритет: безопасность и корректность финансовых расчетов.

## 2. Цель работы агента

- Разрабатывать продукт `AI-финансовый консультант` поэтапно.
- Держать архитектуру чистой: четкие границы слоев и ответственностей.
- Поддерживать качество: тесты, документация, предсказуемые миграции.

## 3. Обязательный процесс для каждой задачи

1. Прочитать релевантные разделы `SPEC.md`.
2. Сформулировать короткий план работ.
3. Реализовать задачу минимально достаточным изменением.
4. Добавить/обновить тесты.
5. Прогнать проверки.
6. Обновить документацию, если изменился контракт/поведение.

## 4. Архитектурные правила

- Backend: Laravel + Clean Architecture.
- Не смешивать доменную логику с инфраструктурой и HTTP-слоем.
- Внешние интеграции (OpenAI, YooMoney) только через абстракции/адаптеры.
- Денежные операции, кредитный ledger, списание баланса выполнять транзакционно.
- Webhook-обработчики платежей делать идемпотентными.
- Любые операции AI, влияющие на данные пользователя, выполнять только через явные разрешенные инструменты.

## 5. Правила для frontend

- Стек: Vue.js (JavaScript).
- Подход: mobile-first, затем desktop.
- Соблюдать дизайн-направление из `SPEC.md` (Liquid Glass, темы, цветовая палитра).
- Не ломать доступность: корректные фокусы, контраст, семантика.

## 6. Правила для backend

- Стек: PHP 8.5, Laravel, MySQL 8.
- Валидация входных данных обязательна.
- Ошибки API отдавать в единообразном формате.
- Версионировать API при breaking changes.
- Для новых модулей создавать миграции, сиды и минимальный набор feature-тестов.

## 7. Безопасность и комплаенс

- Минимизировать хранение персональных данных.
- Не логировать секреты и чувствительные данные.
- Учитывать 152-ФЗ и требования к обработке платежей.
- Все секреты хранить в переменных окружения, не в коде.

## 8. Работа с LLM и стоимостью

- Логировать модель, токены (input/output/cache), стоимость и связь с пользователем/диалогом.
- Перед AI-вызовом проверять лимиты и доступный кредит.
- После AI-вызова списывать стоимость через ledger, не через “голое” поле баланса.
- Поддерживать конфигурируемые модели и тарифы токенов.

## 9. Тесты и проверки

- Backend: unit-тесты для доменной логики.
- Backend: feature-тесты для API и бизнес-критичных потоков.
- Frontend: unit-тесты для ключевой логики.
- Frontend: e2e-тесты для критических пользовательских сценариев.
- Минимум перед завершением задачи: тесты измененных модулей должны проходить.

## 10. Формат изменений

- Делать небольшие, логически завершенные изменения.
- Не выполнять массовые рефакторы без явной необходимости задачи.
- Не менять публичные контракты без обновления документации и плана миграции.
- Для новых решений фиксировать ADR в `docs/adr/`.

## 11. Предпочтительная структура проекта

- Использовать структуру из раздела “Предлагаемая файловая структура репозитория” в `SPEC.md`.
- Новые файлы размещать согласно доменам: `Finance`, `Budget`, `Subscription`, `AI`, `Notification`, `Support`, `User`.

## 12. Что агент должен сообщать по итогу задачи

- Что сделано.
- Какие файлы изменены.
- Какие проверки запущены и их результат.
- Какие риски/ограничения остались.
